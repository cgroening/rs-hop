# CLAUDE.md – hop (Rust)

Context for Claude Code sessions in this directory. It carries what you need to keep developing the project **and** the binding coding rules (so the external style guide need not be opened each time).

## 1. What this is

`hop` is a fuzzy-finder TUI in **Rust (edition 2024)** to jump between git repositories and folders. It is a port of the Python `git-repo-jumper`: pick an entry, its path is written to a handoff file the shell wrapper reads to `cd`, and (for git repos, on TUI `L`/`l`) the configured git tool (lazygit) can be launched. On top of the original it adds in-app management, two kind-tabs each with its own archive, per-kind sections, slugs for a `hop <slug>` fast jump, sort modes, and a marker plus picker for paths that no longer exist.

Main screen: **ratatui** + **crossterm** TUI with two tabs (Git Repos / Files and Folders), each reachable in an active or an archive state (a second press of `1`/`2` toggles the archive), a live fuzzy filter, and modal overlays (confirm, slug prompt, add/edit form, path picker, help).

## 2. Invariants and key behaviours

- **XDG paths, replicated by hand** in `src/util/paths.rs` (the `directories` crate resolves differently on macOS): config `$XDG_CONFIG_HOME/hop/config.toml`, state `$XDG_STATE_HOME/hop/` for the git-info cache (`git-info-cache.toml`), usage state (`usage.toml`), the selected-repo handoff (`selected-repo.txt`) and the log (`hop.log`). `~` is expanded in config paths. Home prefers `$HOME`, then `%USERPROFILE%`.
- **Config = settings, repository = entries, same file.** `config::loader` reads only the settings (`git_program`, `github_username`, `example_mode`, `editor`, `editor_extensions`, `zip_backup_folder`, `zip_exclude_dirs`, `[icons]`, `[column_widths]`) and ignores `[[repos]]`; `storage::toml_repo_repository` reads/writes the `[[repos]]` array. Both point at the resolved config path. Writing entries back goes through `config::writer` (`toml_edit`), so the settings block and its comments survive; the `[[repos]]` array is rebuilt (per-entry comments are not preserved). Runtime fields (live git info, usage counters) are never persisted.
- **Tabs = kind × archived.** `domain::filter::Tab` is a 4-variant enum (`GitActive`, `GitArchive`, `FilesActive`, `FilesArchive`) over a `TabKind` (`Git` | `Files`); `belongs_to_tab` matches an entry's `kind` and its `archived` flag. `1`/`2` (`Action::TabGit`/`TabFiles` → `App::select_kind`) switch kind, or toggle a kind's active↔archive when it is already showing; `Tab`/`Shift+Tab` cycle only the two active tabs (`Tab::ACTIVE`); the header suffixes the active tab with ` · Archiv` (`appframe::TabBar`). `Tab::as_key`/`from_key` persist all four (legacy `"archive"` → `GitArchive`). An entry's `archived` bool (persisted per `[[repos]]`) moves it to its kind's archive; its `section` is never touched by archiving, so the group is preserved.
- **Sections (both kinds, per-kind namespaces).** Each entry has an optional `section` (a `Repo` field, persisted per `[[repos]]`). There are **two** independent section-name lists: `git_sections = [...]` and `sections = [...]` (files; the legacy key, so old files-only configs still load) in `config.toml`. The `RepoRepository` trait's `find_sections`/`save_sections` take a `RepoKind`; `config::writer` key-parameterises the array. The service holds both lists and every section method (`sections`/`add_section`/`rename_section`/`delete_section`/`move_section`/`ensure_section`) takes a `RepoKind`; rename/delete rewrite **only same-kind** entries. `domain::sections` is the pure grouping (`group`, `flatten`, `section_starts`, `current_section`, and the `UNGROUPED` label). Grouping is a **per-tab toggle** (`App::grouped`, default on): when on, `App::is_sectioned` routes to `tui::sections_view` (a `List` of full-width header bars + entry rows, scroll offset carried in `App::list_offset`); when off, `tui::table` renders a flat, globally sorted list with a leading Section column (`TableView::show_section`). Within a group entries follow the active sort mode (`section_groups` sorts the flat index list, then `sections::group` keeps that order per bucket). `s` jumps to a section, `Alt+↑/↓` reorders within a group (custom sort only), `M` opens the manage-sections overlay (`tui::sections_modal`, `Overlay::Sections` + `SectionPrompt`/`SectionDelete`, operating on the current kind's list); "Ungrouped" is virtual and locked. `,` (`Action::ToggleFavFloat`) and `.` (`Action::ToggleGrouping`) toggle fav-floating and grouping per kind. The sectioned view renders git columns (branch/status/GitHub/ZIP) for the git kind via the shared `tui::git_columns` helpers + `presentation::status_span`; the git table uses the same helpers.
- **Auto-naming**: an entry's `name` is optional; `Repo::display_name` falls back to the path's basename.
- **Runtime hydration**: usage counters are loaded onto entries in `RepoService::new`; git status is loaded from the cache at TUI start and refreshed in the background (`service::status_service::spawn_refresh`, one worker thread streaming `StatusUpdate`s over an mpsc channel - a `Started { path }` before each repo and a `Done { path, info }` after, so the progress bar can name the repo while git runs). The run loop polls with a 150 ms timeout and drains updates, so the list shows immediately and fills in. `example_mode` shows each entry's `example_git_info` and skips all git calls.
- **Fetch / startup status**: `tui::StartupStatus` (`Cached` | `Refresh { fetch }`) decides what `App::new` does on start; `cli::startup_status` derives it from `--cached` (cache only, no git), `--fetch` and the `fetch_on_start` config key (`fetch = --fetch || fetch_on_start`). In the TUI `R` = `git fetch` + reload all, `r` = reload all only (`reload_status`); `x`/`X` = refresh only the selected entry without/with fetch (`refresh_one`, which leaves the global `fetched_at` untouched). `hop <slug> --fetch` fetches the repo before the jump (refreshing remote refs). The cache is rewritten from the full entry state on every refresh finish, so a single-entry refresh never drops the others. `GitClient::fetch` runs `git fetch --quiet` and must capture output / skip missing paths (otherwise git's stderr corrupts the alternate screen).
- **Header / status bands**: the header band carries only the brand + tab bar (`appframe::render_header` over `ratada::tabs`, active tab in `Colors::tab_active`, the accent's hue complement). Everything status-like sits in the **bottom status band** (`App::status_lines`): the info line (error count, entry count, sort, local status time and remote fetch time, each behind an `IconSet` glyph, in `Colors::muted`), plus the live-filter input or a transient message. The remote segment warns (amber `Colors::changes`) when over 24h old or never fetched. `fetched_at` is persisted in the git-info cache (`status_service::save_cache(.., fetched_at)`), updated only when a refresh actually fetched (`App::refresh_fetched`).
- **Progress bar**: while `loading`, a single row above the status band (`appframe::FrameAreas::progress`) shows `ratada::gauge::render` filled to the ratio, its centred label from `ProgressText::label`: a refresh reads `XX % - <name>` (the entry from the `StatusUpdate::Started` event, set in `drain_status`), a ZIP backup `XX % (<file>/<files>) - <name>`. The percentage is padded to a fixed width and the name to `App::loading_name_width` (the widest name of the run, computed at loading start), so the constant label width keeps the `XX %` column from shifting as names come and go. During a refresh, rows not yet updated this pass show an animated spinner in the status column (`TableView::spinner`, fed by `App::spinner_frame` from a `ratada::spinner::Spinner` that `App::tick` steps every `SPINNER_INTERVAL`).
- **Entry errors**: `Repo::entry_error` (pure) reports a missing path or an invalid git entry. The displayed error is gated by kind through `App::path_error`: **git** entries are flagged live (via `entry_error`, driven by the background refresh); **`Path`** (file/folder) entries only after an existence check – `check_files_existence` stats every `Path` entry and records the missing ones in `App::files_missing`. It runs on the **first visit** to the Files tab (via `refresh_tab_on_first_visit`, including at startup when that tab is active, skipped in `example_mode`) and again on every `r` there; it is never run continuously/live. The marker `!` (in `tui::table` for git/archive and `tui::sections_view` for files, both fed `&files_missing`), the header error count and the `!` error list all use `path_error`. `!` opens the error list (`Overlay::Errors`); selecting one opens an action menu (`Overlay::ErrorAction`: repair path / edit / delete) reusing the index-parameterised `repair_picker_for`/`edit_form_for`/`delete_confirm_for`.
- **Kinds and opening.** `RepoKind` is `Git | Path` (the legacy `folder`/`file` config values fold into `Path`; `as_config_value` writes `"path"`). A `Path` entry is told apart at open time by `domain::repo::classify_path(path, &editor_extensions) -> PathClass { Folder, TextFile, OtherFile }`: an existing dir (or a non-existent path written with a trailing `/`, via `is_dir_target`) is a Folder; otherwise a file split by `is_text_file` (no extension, or an extension in `Config::editor_extensions`, default `DEFAULT_EDITOR_EXTENSIONS`). **Opening** (`tui::open_selected(launch_tool)` -> `RunOutcome`, acted on by `cli` after the terminal is restored): `L` launches the git tool (Git), `cd`s (Folder), opens `$EDITOR` (TextFile -> `RunOutcome::OpenFile`) or the OS default app (OtherFile -> `RunOutcome::OpenWith` -> `util::opener::open_default_app`, macOS `open`); `Enter` and `o` only write the handoff `cd` path (a file `cd`s to its parent), and `l` runs the git tool inline. The add/edit form warns when a `Path` is saved without a trailing `/` and the path does not exist (assumed a file). `hop <slug>` only writes the `cd` path (like `Enter`/`o`, no tool/editor); `--save-only`/`-s` is a deprecated no-op.
- **Slugs** (`domain::slug`) are `[a-z0-9-]+`, validated against reserved command names (`domain::slug::RESERVED`: `add`, `scan`, `doctor`, `list`, `config-path`, `help`) and for uniqueness by the service. A CLI test (`every_subcommand_name_is_a_reserved_slug`) cross-checks `RESERVED` against the clap subcommands so a new command can't be shadowed by a slug.
- **CLI add / scan.** `hop add [PATH]` registers one entry (auto-detecting kind via `resolve_kind`). `hop scan [DIR] [--depth N] [--nested] [--dry-run]` discovers git work trees (`util::scan::find_git_repos`, std-fs walk, prunes hidden/`node_modules`/`target`, no symlinks, stops at `.git` unless `--nested`), drops the already-known ones (`partition_found` on canonical paths, reported as a count) and imports the chosen ones from a `sparcli::Select` multi-select prompt (`cli::pick_repos_to_import`, everything pre-checked) via `RepoService::add_many` (one undo frame); off a TTY `run_multi` returns `SparcliError::NoTerminal` and hop points at `--dry-run` instead of hanging. `hop doctor` reports problems via the pure `domain::doctor::diagnose` (missing path, git entry that is not a repo, malformed slug, duplicate slug) and exits non-zero when any are found.
- **Multi-select / bulk**: `App::selected` (service indices) + `anchor`; `Space` toggles, `Shift+↑/↓` ranges (`extend_selection`), `Esc`/tab-switch clears. `App::targets()` returns the selection or the cursor entry; the mutating bulk keys `d`/`A`/`*`/`x`/`z` act on it via the service bulk methods (`delete_many`/`set_archived_many`/`set_fav_many`, each one undo frame) and clear the selection. `e` (edit) opens a **bulk form** (`RepoForm::for_bulk`) when `targets().len() > 1`: Path/Name/Slug are hidden, Section/Fav/Backup/Kind are shown with a per-field touched flag and a *mixed* marker (`[-]` / `(mixed)`) where the selection differs; `App::do_save_bulk` applies only the touched fields to all via `RepoService::update_many` (one undo frame). `App::shared` computes each field's common-or-mixed value. The form target is `Overlay::Form`'s `EditTarget { Add | One(usize) | Bulk(Vec) }`. The read-only bulk keys `y` (copy every target path, one per line, via `target_paths`) and `b` (open a GitHub tab per target git repo, via `github_url_for`) act on `targets()` too but leave the selection intact. Selected rows get the `colors::MULTI_SELECT_BG` tint (`TableView::selected`).
- **ZIP backups.** `z` zips the target entries (selection or cursor; git repos and folders, non-folder/missing paths dropped by the `is_dir` filter in `start_zip`), `Z` (`zip_all`) zips every entry whose `Repo::include_in_backup` is true across tabs, into `config.zip_backup_folder` (`~` expanded). `include_in_backup` (a `Repo` field, persisted per `[[repos]]`) defaults per kind - git: included, path: excluded - resolved on read (`toml_repo_repository::into_repo`, absent → `kind == Git`) and written by `config::writer` only when it deviates from that default; the add/edit form (`tui::form`) exposes it as the `Backup` checkbox for both kinds, seeded per kind on add. `z` ignores the flag (explicit target); only `Z` honours it. Excluded entries render the `IconSet::excluded` glyph (`⊘`) instead of a date in the `ZIP Backup` column. The file name is unique per entry via `domain::backup::backup_filename` (pure): the slugified `display_name` (`slug::slugify`), plus a stable FNV-1a path-hash suffix when two entries (git or folder) share a base name (so no archive is overwritten); `domain::doctor` flags the clash as `DuplicateBackupName`. `tui::mod::backup_dest` joins it to the folder and is the single source for both the zip job and the column date lookup. `util::archive` is the pure-ish layer (`collect_files` walks std-fs pruning any dir whose name starts with an excluded prefix via `is_excluded` (so `target` covers `target.nosync`), keeping `.git`; `write_zip` deflates with the `zip` crate, top-level folder preserved, per-file progress). `service::zip_service::spawn_zip` is the background worker (mirrors `status_service`): counts files for the total, then per repo compares a content fingerprint (`archive::source_manifest`, name+size+CRC32 via `crc32fast`) against the existing archive (`archive::zip_manifest`, read from its central directory) and **only rewrites when the content changed** — so an unchanged repo's `.zip` is left untouched (no needless iCloud re-upload). It streams `ZipUpdate { done, total, finished, archives, unchanged, errors }` over an mpsc channel. The unchanged-check is short-circuited by a local fingerprint cache (`storage::zip_cache`, `zip-manifests.toml` in the state dir, keyed by archive path → content hash + the dest's mtime/size), so an iCloud-evicted archive is not re-downloaded when the file on disk is still the one we wrote; it only reads the archive when the cache is missing or the file changed externally. `archive::write_zip` builds into a `<dest>.part` temp and atomically renames over `dest` (so a failed build never corrupts the previous backup), and preserves Unix permissions and symlinks (stored via `add_symlink`, whose target content the fingerprint accounts for). `App::start_zip`/`drain_zip` reuse the progress bar above the status band (`App::loading` + `loading_label`, "zipping" vs "refreshing"; the label picks the refresh vs zip text format in `App::progress_prefix`, `render_progress` hands a `ProgressText` label to `ratada::gauge`); a zip is refused while another refresh/zip runs (`loading.is_some()`). All three tabs show a `ZIP Backup` column with each entry's last-backup date: the Git Repos and Archive tabs via `tui::table` (`column_widths.zip_backup`), the Files tab via a fixed-width right-aligned column in `tui::sections_view` (`ZIP_WIDTH`). Both source from `App::zip_backups` (a path→time map filled by `reload_zip_backups` at start and after each zip, now covering folder entries too; no per-frame stats) and show the `excluded` glyph for entries opted out of `Z`.
- **Custom order / reorder**: `SortMode::Custom` keeps the stored `[[repos]]` order (favourites floated on top, stable). `Alt+↑/↓` (`move_entry`, only in custom sort) swaps the cursor entry with its neighbour within the same favourite group via `RepoService::swap_entries`, which persists the new `[[repos]]` order. Footer hints are grouped by kind: `tui::bindings::hint_groups(tab)` keys on `tab.kind()` (both kinds share `Open`/`Manage`/`Backup`/`View`/`Panel`/`Sections`; git adds a `Git` refresh group, files a `Paths` group and drops `l`), and `App::hint_groups` turns them into `(key, description)` pairs via `config.keymap().hints(...)` behind a compact static `Navigation` group. Per-kind view settings (sort mode + direction, column set, grouping, fav-float) live in `App::tab_state: [TabState; 2]` (indexed by `Tab::kind_index()`, accessed via `App::view()`/`view_mut()` and the `sort()`/`columns()`/`grouped()`/`fav_float()` accessors), and are persisted across runs in `$XDG_STATE_HOME/hop/ui-state.toml` as `[tabs.git]`/`[tabs.files]` blocks; the active tab, slug-column toggle, preview-panel mode and hint visibility are top-level globals (`storage::ui_state::{UiState, TabView}`; a legacy top-level `sort`/`columns` seeds both kinds). Loaded in `App::new`, written together via `save_ui_state` so no field clobbers another. `Tab::as_key`/`Tab::from_key` map the tab to/from its stored string. `i` toggles `App::show_slugs`: when on, the slug is shown in its **own dim-italic column** right after the name (`presentation::slug_style`), content-sized, in both `tui::table` (with a `Slug` header, next to the optional `Section` column) and `tui::sections_view`; the name column no longer carries the slug.
- **Missing paths**: the table shows a red `!` in the leftmost column; `p` opens the path picker at the nearest existing ancestor (`domain::path_repair::nearest_existing`).
- **Navigation / misc keys**: `Tab`/`Shift+Tab` cycle tabs (`cycle_tab`); `g`/`G` jump to the list ends, `PageUp`/`PageDown` and `Ctrl+u`/`Ctrl+d` page/half-page (`page`, using `App::list_height` set during render); `u` runs `RepoService::undo`; the cursor entry's path is remembered per tab (`App::tab_focus`) and restored on return (`remember_focus`/`restore_focus`). `b` opens the entry on GitHub (`presentation::github_url` + `util::opener::open_url`, github.com assumed, non-blocking so the TUI stays up). Sorting adds **frecency** (`SortMode::Frecency`, `open_count` weighted by recency decay; `sort_indices`/`sort_repos` take `now: i64`). The header info line shows `shown/total` plus active-lens markers (filter / changes / slugs) when narrowed. Fuzzy matches are highlighted in the name (`presentation::highlight_name`, only in `tui::table` since filtering disables the sectioned view).
- **Detail panel** (`tui::preview`): `PreviewLayout { visible, position, width_pct, height_rows }`, persisted. `v` toggles it, `V` flips right/bottom, `Ctrl+↑/↓` scrolls (a shared `tui::scroll::Scroll`), `Ctrl+←/→` resizes along the current axis. `split_preview` leaves a one-cell gutter (`PANEL_GUTTER`) and uses **separate** minimums per axis (`MIN_LIST_COLS` vs `MIN_LIST_ROWS`). The panel never `Clear`s its area - `appframe` already tinted it, and clearing punched a hole to the terminal default. Its title comes from `ratada::chrome::border_title` (`╭─ Details - name ───`), its rows from `tui::detail` (`section_header`/`field`/`log_line`/`bar`), its scrollbar from `ratada::scroll::render_scrollbar`. Sections: identity, CODE (from the stats cache), GIT (live status + history), LOG (lazily fetched via `App::request_preview_log`).
- **Project statistics.** Two **independent** families, because the `Code` columns never need git and `Activity` never needs a source walk: `service::stats_service::spawn_code_stats` (tokei + `util::diskusage::measure`) and `spawn_git_stats` (`GitClient::stats`). Both mirror `status_service` (one thread, mpsc, stops when the receiver drops). Progress is **per entry**, not per file: tokei's `for_each_fn` is a bare function pointer and cannot capture a channel. Results land in `storage::stats_cache` (`stats-cache.toml`, two independent tables) which is a **display accelerator only** - there is no invalidation; the cache seeds the view and the worker overwrites it. `tokei::Config::default()` already skips `.git` (hidden off) and honours `.gitignore` (`no_ignore_vcs` off); `no_ignore_parent` is turned on. `zip_exclude_dirs` are translated to directory globs (`target` → `target*/`) and layered on top - the integration test `tests/code_stats.rs` pins that translation, **including** the fact that `.gitignore` is only read inside an actual repository (a plain folder entry relies on the exclusions alone). `App::start_stats` runs nothing in `example_mode` and nothing for `ColumnSet::Standard`; unresolved cells then read `-`, never a spinner.
- **Column sets** (`tui::columns`, `c`, persisted): `Standard` (as before), `Code`, `Activity`. The Files tab omits `Activity` (`ColumnSet::all`), and a persisted set the tab lacks falls back via `available_on`. Outside `Standard`, `render_footer` draws a column-aligned totals row over the **visible, filtered** entries plus a tinted bar (`Columns  Standard › Code › Activity`); `footer_rows` returns 0 below `MIN_CONTENT_ROWS` so a short terminal keeps its list. `cell_text` returns `None` while a worker owes the value, which is what the renderers turn into a spinner. `tui::sections_view` draws a column-title row (via `render_column_header`/`header_spans`, sized with the same width helpers as the rows so it aligns) plus a blank spacer **above** the list for **every** set (git/files Standard and the statistics sets), so `settled_offset` keeps working unchanged.
- **Sorting** (`t`): `SortMode` grew the column modes and a separate `SortDir`; `sort_indices` takes a `SortContext { mode, dir, now, stats }`. The statistics modes do not pin favourites, and an entry whose number is not computed yet sorts **last in both directions** so the order does not jump while workers report. `SortMode::next` and `sort_repos` were removed (no callers, and the latter could not sort by half the modes).
- **One-level undo** is in the service (`RepoService` records the pre-change entry list per config mutation and rolls back if a write fails); the TUI binds it to `u` (`Action::Undo`).
- **Atomic writes.** Every config and state write goes through `util::fs::write_atomic` (sibling `.tmp` + rename), never `fs::write` – the settings and all `[[repos]]` share one file, so a truncating write that fails midway would lose the entries.

## 3. Architecture (layered; dependencies point inward)

`domain` ← `storage` ← `service` ← (`cli`, `tui`); `config` and `util` are leaf utilities. `main.rs` is the composition root; a **library crate** (`src/lib.rs`) holds all logic so the integration tests in `tests/` drive the public API. DIP via the `RepoRepository` and `GitClient` traits, injected as `Box<dyn …>` / `Arc<dyn …>`.

**`src/tui/` must not name a storage adapter** (only `cli` may, as the composition root). The state files it reads and writes come through the service owning the work that produces them: `status_service::{load_cache, save_cache}`, `stats_service::{load_cache, save_cache}` and `ui_state_service::{load, save}`, each re-exporting the state type it hands out. The one `use crate::storage::…` left in the TUI is the `GitClient` **port** – a trait, injected. hop uses a single `domain::error::Error` across domain/storage/service (no `storage/errors.rs`, no `From` funnel); that is a deliberate deviation from clibase, recorded in `docs/DEVELOPMENT.md`.

### Migration to clibase (ratada/sparcli, panel layout)
The app was migrated onto the `clibase` template. Key structural additions: `src/keymap.rs` (the `Action` catalog + `Keymap::from_overrides` SSOT for `[keys]` bindings) drives **both** the grouped footer hints via `src/tui/bindings.rs` (per-tab `Action` groups → `keymap.hints`) **and** the TUI key dispatch: `App::handle_list_key` resolves the key with `Keymap::action_for` and matches on the `Action` in `App::run_action`, where the tab-dependent cases (`Sort`, `Reload`/`ReloadFetch`, `ManageSections`) branch on the tab rather than the key. The map is built once into `App::keymap`. `App::handle_untracked_key` consumes the three keys the keymap cannot express first: `Shift`+arrow (`KeyChord` ignores shift, so it would resolve to a plain cursor move), `Tab`/`BackTab` and `Esc`. `Ctrl+Q` never reaches dispatch (the `Tui` turns it into `TuiEvent::Quit`). Further: `src/config/appearance.rs` (`[appearance]` theme/colours/glyphs), `Config` accessors `palette()`/`skin()`/`theme_registry()`/`color_overrides()`/`keymap()`, plus `[themes.<name>]` custom themes over the built-in rose `default`/`monochrome`. `src/tui/appframe.rs` draws the panel frame (`render_frame -> Rect`: tinted borderless header/content/status bands + backgroundless `shortcut_hints`; `snapshot_frame`/`dim_backdrop` give modals a dimmed live-view backdrop). `src/tui/skin.rs` (`Colors::from_palette`) bridges the runtime palette to hop's colour roles, and **every** renderer draws from it: the table and sections view carry it on their `TableView`/`SectionedView` context and `App` resolves it once into `App::colors`. No role is a compiled-in colour (a test pins that a monochrome theme leaves nothing coloured), so re-theming recolours the content cells, not just the frame and modals. Overlays keep hop's **non-blocking** state machine (background refresh keeps running) but are restyled with `ratada::chrome::modal_block` + the dimmed backdrop, rather than swapped for `ratada`'s blocking `modal::*`; the help overlay (`tui/help.rs`) is likewise hand-drawn but styled to match `ratada::help` (uppercase section headers, aligned key column, hint footer), with a trailing `Global` section from `tui::mod::global_group`. The one blocking `ratada` dialog is the quit confirmation (`ratada::quit` guard + `modal::confirm`), which is safe because nothing needs to drain after a quit; `Ctrl+Q` bypasses it. `F1` toggles the hint footer via `ratada::shortcut_hints::consume_toggle`, consumed at the top of `App::handle_key`, and its state persists in `UiState::hints_visible`. The add/edit form keeps its own `RepoForm` (path picker, kind/section cycles, inline slug validation), restyled as a **centred modal** rather than moved inline into the content panel. The **path picker is hop's own `PathPicker`**, not `ratada::path_picker` (which is blocking). Inside those shells everything is the toolkit's: text fields are `ratada::input::InputField` painted by `presentation::field_spans`, lists are `ratada::list`, badges `ratada::chrome`. Both deviations (non-blocking overlays, modal placement of forms) are recorded in `docs/DEVELOPMENT.md` (Architecture decisions).

## 4. Where to make common changes

- **New list key / behaviour**: add the action to `src/keymap.rs` (`Action` variant + `ACTIONS` row with its default keys), handle it in `App::run_action` (`src/tui/mod.rs`), place it in a group in `src/tui/bindings.rs` (the footer), and keep the `help::SECTIONS` list and `README.md` in sync.
- **New overlay**: add an `Overlay` variant + its state struct (model the existing widgets), handle it in `handle_overlay_key` and `render_overlay`. Box the variant if it is large (as `Form` is).
- **New entry field**: extend `domain::repo::Repo`, then the (de)serialize in `storage::toml_repo_repository` / `config::writer` (keep them in sync), the form in `tui::form` (+ `RepoDraft` and `apply_draft` in `tui::mod`), and both renderers (`tui::table` for the flat view, `tui::sections_view` for the grouped view; shared git-column text lives in `tui::git_columns`).
- **New config setting**: `config/mod.rs` (field + default) + `config/loader.rs` (resolve, honouring any `HOP_` env override; booleans through `parse_bool`, which ignores a typo rather than reading it as `false`).
- **New state file**: adapter in `storage/`, written via `util::fs::write_atomic`, and reached from the TUI only through a service in `service/` that re-exports its type.
- **New text field**: a `ratada::input::InputField`, painted with `presentation::field_spans` (never a hand-rolled caret - it would lose the horizontal scrolling and the `…` marker).
- **Form's Section field** is chosen via the fuzzy `tui::section_picker` (opened by `Enter` on the field → `FormResult::PickSection`, round-tripped through `Overlay::SectionPicker` which re-opens the form via `set_section`); it holds the value as `RepoForm::section: Option<String>` (no cycle), rendered bracket-less (`value_line`), and a typed-but-new name is registered on save via `ensure_section`.
- **New CLI command**: add a `Command` variant + handler in `src/cli/mod.rs`.
- **Git status detail**: `storage::subprocess_git_client` gathers it; `tui::presentation::status_text` formats it from the structured counts.

## 5. Conventions specific to this repo

- Rust edition **2024**; `rustfmt.toml` sets `max_width = 80` (plus nightly-only import grouping). Write imports std → external crate → `crate`, one per line.
- Build artifacts go outside the repo (`.cargo/config.toml` sets `target-dir`) to keep them out of iCloud.
- **Never commit.** At the end of a change, propose one imperative English commit-message title only.
- TUI is verified manually in a real terminal; the pure layers carry the tests.
- **Glyphs: two tiers only - `unicode` (default) and `ascii`** (`[appearance].glyphs`, `ratada::theme::GlyphVariant` → hop's `IconSet`; the legacy `[icons].variant` still loads). No Nerd Font glyphs and no colourful emoji: a Nerd Font icon sits in the Private Use Area where `unicode-width` reports one cell but many terminals render two, which clipped the column and made markers render at half width. Every glyph in `IconSet` must be a single-cell symbol.
- **Reach for the toolkit before writing a widget.** `ratada` owns text editing (`input::InputField`), the block caret, `nav::cycle`, `text::truncate`, `list`, `scroll`, `gauge`, `spinner`, `chrome` badges, `clipboard` and `quit`; `sparcli` owns the CLI's tables, alerts and prompts. hop keeps its own **overlay shells** only (see `docs/DEVELOPMENT.md`, Architecture decisions).

# Coding rules (binding)

These reproduce the project style guide (general + Rust). On conflict the more specific (language) rule wins; these documented rules take precedence over rustfmt/clippy defaults.

## G. General (all languages)

- **KISS / YAGNI**; **DRY** (code) and **SSOT** (data); **consistency**; no magic numbers/strings (named constants).
- **SLAP** – one level of abstraction per function. **Max two nesting levels** (early returns / guard clauses). **No flag arguments** – split into two well-named functions. Command–Query separation; prefer pure functions.
- Polymorphism over type switching; composition over inheritance; Tell-Don't-Ask; **SOLID** (DIP via injection); high cohesion, loose coupling.
- **Naming**: booleans as yes/no questions (`is_`/`has_`/`can_`/`should_`, never negative); methods = verbs, types = nouns, no `Manager`/`Data`/`Helper`; `snake_case` vars/functions, `UPPER_CASE` consts, `PascalCase` types; files `snake_case`. Acronyms as words (`UserId`).
- **Errors**: fail fast; don't signal "nothing" with `null`/`None` where an empty collection / error fits; errors carry context (what/where/why). Validate at trust boundaries; fail loudly over silent-wrong.
- **Logging** via the `log` facade, never `println!` for diagnostics (visible CLI/TUI output is not logging).
- **Formatting**: 4-space indent, **max 80 columns** (`.rs`); operators at line start on wraps; trailing commas in multi-line; spaces only; one final newline; UTF-8/LF. **Never use `–` (em dash) anywhere.** As the dash (Gedankenstrich): in code files use `-` (hyphen), in `.md` files use `–` (en dash). Markdown prose: one line per paragraph/item, no hard wraps. Straight quotes only.
- **Comments** explain the **why**, moderately; everything in **English**.
- **Types**: `bool` not int flags; enums over magic strings; structs over loose tuples; newtypes where useful; immutable where sensible; don't mutate inputs needlessly.
- **Functions** small (SRP); **few parameters (≤3, group related into a struct)**; pass named/explicit; helpers below callers; public before private.
- **Architecture**: layer by responsibility; DI through constructors/parameters; depend on abstractions; composition root in `main`. Prefer stdlib; **agree before adding a dependency**.
- **Tests** always ship (FIRST; one concept each; fakes over mocks; names describe behaviour). **Run all tests after every change.** Measure before optimising.
- **Maintenance**: respect local style; minimal focused changes; keep refactoring separate from behaviour changes; update call sites, tests and docs; remove dead code; fix root causes.

## R. Rust-specific

- **Edition 2024**; rustfmt with import grouping; **clippy `-D warnings` must pass**; `log` for diagnostics.
- Submodules via `mod`; abstract behaviour with **traits**, inject via generics or `dyn`.
- **Errors**: `Result<T, E>` + `?`; **`thiserror`** for the domain error enum (one per crate/domain, `#[from]` for foreign errors); **`anyhow`** only at the binary edge with `.context()`; **`unwrap()` forbidden, `expect()` only at provably-infallible points** with a reason; **no `panic!` in normal flow**; avoid `unsafe`.
- **Docs**: `///` on every public item (one-sentence first line, prose, no `# Arguments` lists); `# Errors`/`# Panics`/`# Safety` where they apply; `//!` module summary; private items get a short `///`.
- **Idioms**: enums for states; derive sensible traits; `new()` (+ `Default` where parameterless); prefer borrows, avoid needless `clone`; `Option<T>` for absence; `match`/`if let` with early returns; iterators for simple map/filter; keep visibility minimal, re-export via `pub use`.
- **Concurrency**: sync by default; `async` only for real I/O concurrency; prefer ownership/channels over shared locks.
- **Tests**: unit in-file under `#[cfg(test)] mod tests`; integration in `tests/`; doctests must run.
- **Security**: choose `checked_*`/`saturating_*` for external integer values; `Command` with `.arg()`, never a shell string; validate external paths; no secrets in code/logs; bound untrusted input sizes.

## Git

- **Do not create commits yourself.** At the end of a change, propose a commit-message **title only**, in English imperative, Conventional-Commits style.
- **Maintain `CHANGELOG.md`** (Keep a Changelog format, SemVer). For any user-facing change, add a bullet under the `## [Unreleased]` section in the right group (Added / Changed / Deprecated / Removed / Fixed / Security). On a release, rename `Unreleased` to the new version with the date and update the compare links. Purely internal changes (tests, refactors, docs) need no entry.
